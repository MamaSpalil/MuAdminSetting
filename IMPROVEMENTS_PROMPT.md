# Промт: улучшение функциональности и визуала MuAdmin

Этот документ — постановка задачи и пошаговый план работ по улучшению
программы **MuAdmin** (графический редактор серверных конфигов MU Online).
Документ предназначен для использования как ТЗ для последующих коммитов /
сессий: каждая фаза самодостаточна, имеет критерии готовности и не ломает
обратную совместимость с уже существующими файлами сервера (round-trip
сохраняется — см. `src/MuAdmin.Core/Parsers/*`).

> **Важное замечание об авторских правах.**
> Официальные спрайты монстров, иконки предметов и скриншоты локаций
> MU Online являются собственностью Webzen Inc. Их нельзя «скачать из
> открытого доступа» и закоммитить в этот репозиторий — это нарушение
> копирайта. Поэтому архитектура визуализации построена так:
> программа ищет картинки в локальной папке `assets/` рядом с исполняемым
> файлом (или указанной пользователем). Пользователь кладёт туда свои,
> легально полученные изображения (например, экспортированные из
> собственного клиента). Если файла нет — рисуется аккуратный
> плейсхолдер. Это обеспечивает и легальность, и полноценный визуал
> у конечного пользователя.

> **Обновление (поставка ассетов рядом с конфигами).** В дополнение к
> папке `assets/` менеджер ассетов (`MuAdmin.Core.Assets.AssetsManager`)
> теперь читает картинки прямо из корня сервера:
> - `Maps/<MapName>.jpg|png` — скриншоты локаций (имя файла =
>   название карты из `MapNames`, напр. `Lorencia.jpg`, `Devias.jpg`).
> - `Monster/Monster<NN>.bmd` — оригинальные 3D-меши клиента
>   (двух- и трёхзначные имена). Поскольку BMD — это бинарный
>   формат 3D-моделей, рендерить его в `System.Drawing` без
>   3D-движка нереалистично; программа использует наличие BMD как
>   индикатор «модель есть», а в гриде / превью отображает
>   плейсхолдер `M:N`. Если рядом с `.bmd` положить одноимённый
>   `.png`/`.jpg` — менеджер подхватит его автоматически.

## 1. Исправление ошибок табуляции и проверка столбцов

Файлы сервера MU — это в основном whitespace-разделённые таблицы с
комментариями (`//`) и маркерами `end`. Сейчас `TabularEditor` показывает
безымянные столбцы `c1..cN`, что затрудняет редактирование и
провоцирует ошибки.

### 1.1 Схемы столбцов
- [ ] Создать `src/MuAdmin.Core/Schemas/TabularSchema.cs` —
      описание схемы файла: имя файла (glob), список колонок
      (`Name`, `Type`, `Description`, `Required`, опц. `EnumValues`).
- [ ] Создать `src/MuAdmin.Core/Schemas/SchemaRegistry.cs` со
      встроенными схемами для основных файлов:
  - `Monsters/Monster.txt`
  - `Monsters/MonsterSetBase*.txt`
  - `Items/Item.txt`, `Items/ItemAddOption.txt`
  - `Quests/Quest.txt`, `Quests/QuestRewardSetting.txt`
  - `Move/MoveReq.txt`
  - `Shops/Shop_*.txt`
  - `EventItemBags/*.txt`
- [ ] Источник истины — официальная документация формата серверов
      (комментарии в шапке каждого `.txt`). Парсить шапку и, если
      встроенной схемы нет, выводить колонки из неё.

### 1.2 Применение схемы в редакторе
- [ ] В `TabularEditor.BuildTable` использовать схему: задавать
      `Caption` колонкам, `ToolTipText` с описанием, `MinimumWidth`,
      `DefaultCellStyle.Format` (числа — выравнивание вправо).
- [ ] Колонки сверх схемы помечать как `Extra` и подсвечивать —
      это типичный признак ошибки табуляции.
- [ ] Колонки до конца схемы, но пустые — подсвечивать жёлтым,
      если поле `Required`.

### 1.3 Валидация
- [ ] Добавить `TabularValidator` (Core): проверяет
  - количество колонок в каждой data-строке против схемы;
  - типы (`int`, `hex`, `float`, `bool`, `enum`);
  - диапазоны (если заданы в схеме);
  - ссылочную целостность (например, `MonsterIndex` в
    `MonsterSetBase` существует в `Monster.txt`).
- [ ] Панель «Проблемы» внизу редактора со списком предупреждений и
      двойным кликом — переход к ячейке.

### 1.4 Визуальное выравнивание
- [ ] Использовать моноширинный шрифт (`Consolas, 9pt`) в `DataGridView`,
      авто-ширину по содержимому (`AutoSizeMode = AllCells`) с
      разумными пределами.
- [ ] Чередующаяся подсветка строк (`AlternatingRowsDefaultCellStyle`).
- [ ] Включить «заморозку» первых N колонок (по схеме — обычно
      `Index` + `Name`).

**Готово, когда:** все встроенные файлы открываются с понятными
заголовками, лишние/пропущенные колонки видны визуально, валидатор
выдаёт ≤ известных проблем на чистых данных, round-trip сохраняется
(тест: открыть → сохранить без правок → `git diff` пуст).

## 2. Инфраструктура ассетов (фундамент визуализации)


## Критерии «профессионально выполнено»

- Никаких регрессий round-trip (см. memory: `OriginalText`/`Modified`).
- Никаких копирайт-нарушающих ассетов в репозитории.
- Все новые публичные классы/методы имеют XML-doc на русском в стиле
  существующих (`/// <summary>`).
- Соответствие целям проекта: `.NET Framework 4.8`, `LangVersion 7.3`,
  явные `<Reference>` в `.csproj`, без новых тяжёлых зависимостей
  (для картинок достаточно `System.Drawing`).
- Каждое UI-изменение проверено вручную на реальных файлах из
  корня репозитория (`Monsters/`, `Quests/`, `Items/`).
